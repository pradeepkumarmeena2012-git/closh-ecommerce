# CLOSH Payment Flow Documentation (Razorpay Integration)

This document explains the "Prepaid" payment flow implemented for the CLOSH platform.

## 1. Flow Overview

The platform supports **Prepaid**, **Cash on Delivery (COD)**, and **Digital at Door** payment methods. The Prepaid option is powered by Razorpay.

### Sequence of Events (Prepaid)

1.  **Selection**: User selects the "Prepaid" option on the `/payment` page.
2.  **Order Initiation**: When the user clicks "Place Order", the frontend sends a request to `POST /api/user/orders` with `paymentMethod: 'prepaid'`.
3.  **Backend Preparation**:
    *   The backend validates the cart and inventory.
    *   Creates a new Order record with `paymentStatus: 'pending'`.
    *   Uses the Razorpay Node SDK to create a **Razorpay Order** (`razorpayOrderId`).
    *   Saves the `razorpayOrderId` in the Order document.
    *   Returns the `orderId` (CLOSH ID), `razorpayOrderId`, and `razorpayKeyId` to the frontend.
4.  **Checkout Modal**: The frontend receives the response and immediately opens the **Razorpay Standard Checkout Modal** using `window.Razorpay`.
5.  **Payment Completion**:
    *   User completes the payment in the Razorpay UI.
    *   Razorpay returns a `razorpay_payment_id`, `razorpay_order_id`, and `razorpay_signature`.
6.  **Verification**:
    *   The frontend sends these details to `POST /api/user/orders/verify-payment`.
    *   The backend verifies the signature using the `RAZORPAY_KEY_SECRET`.
    *   Upon successful verification, the Order's `paymentStatus` is updated to `'paid'`.
7.  **Finalization**: The user is redirected to the `/order-success` page.

## 2. API Endpoints

### Place Order
*   **URL**: `/api/user/orders`
*   **Method**: `POST`
*   **Payload**: Includes `paymentMethod: 'prepaid'`.
*   **Response**: 
    ```json
    {
      "success": true,
      "data": {
        "orderId": "CL-XXXXXX",
        "razorpayOrderId": "order_XXXXXX",
        "razorpayKeyId": "rzp_test_XXXXXX"
      }
    }
    ```

### Verify Payment
*   **URL**: `/api/user/orders/verify-payment`
*   **Method**: `POST`
*   **Payload**:
    ```json
    {
      "orderId": "CL-XXXXXX",
      "razorpayPaymentId": "pay_XXXXXX",
      "razorpayOrderId": "order_XXXXXX",
      "razorpaySignature": "hashed_signature"
    }
    ```

## 3. Database Schema Updates

The `Order` model now includes:
- `paymentMethod`: Enum includes `'prepaid'`.
- `razorpayOrderId`: Stores the ID generated by Razorpay for tracking.
- `razorpayPaymentId`: Stores the successful transaction ID.
- `razorpaySignature`: Stores the verified signature (hidden by default).
- `paymentStatus`: Updated to `'paid'` after verification.

## 4. Frontend Integration

*   **Component**: `frontend/src/modules/user/pages/Payment/PaymentPage.jsx`
*   **Logic**: Uses a state-based selection for "Prepaid". The `handlePlaceOrder` function branches into the Razorpay handler if "prepaid" is the normalized method.

## 5. Security Notes

*   **Signature Verification**: Payments are never trusted based on frontend callbacks alone. The `verify-payment` endpoint re-calculates the HMAC signature on the server to prevent spoofing.
*   **Status Integrity**: Orders remain in `pending` status if the payment window is closed or fails.

## 6. Implementation Notes

*   **Store Persistence**: The `orderStore.js` was updated to preserve `razorpayKeyId` and `razorpayOrderId` from the initial `POST /api/user/orders` response. This is critical because while `razorpayOrderId` is stored in the database, `razorpayKeyId` is a configuration value returned only during order creation. Bypassing the store's default normalization ensures the frontend has all required keys to open the Razorpay modal.
